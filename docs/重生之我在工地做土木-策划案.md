# 重生之我在工地做土木 · 游戏策划案

> 文档性质：完整策划方案，含主题映射、游戏机制设计、内容规划、代码重构路径  
> 基线版本：中关村学院 · AI 博士生模拟器（当前代码库）

---

## 一、主题概述

### 1.1 核心设定

你是一名刚毕业的土木工程专业学生，阴差阳错来到了一个大型建筑工地。从最基层的实习生做起，经历搬砖打灰、测量放线、盯施工现场、处理甲方变更……一步步在工地摸爬滚打，最终成长为能独当一面的项目经理（或以各种方式"毕业"离场）。

### 1.2 游戏标题

**重生之我在工地做土木**

### 1.3 基调与风格

- **黑色幽默 + 写实吐槽**：保留原版的诙谐基调，将学术圈段子替换为工地/土木圈经典梗
- **真实感**：施工阶段、安全管理、甲方关系等取材自真实工程行业
- **轻度策略**：资源管理 + 人际关系 + 阶段推进，每个季度的决策都有意义

## 四、特色系统设计（新增/强化）

### 4.1 安全管理系统（替代讲坛合规）

- 每两个季度为一个安全考核周期
- 需累计完成 **6次** 安全相关行动（安全培训 + 现场巡检）
- 首次不达标 → 警告并结转
- 二次不达标 → 被安监站停工整改（退场）
- **安全隐患值**过高会触发各种负面事件（工伤、罚款、停工）

### 4.2 天气系统（新增特色）

- 不同季节有天气修正：
  - **春季**：多雨，土方工程进度 ×0.8，但种树绿化进度 ×1.2
  - **夏季**：高温，体力消耗 ×1.3，每季度强制1次高温停工
  - **秋季**：施工黄金期，所有施工进度 ×1.2
  - **冬季**：寒冷，混凝土养护困难，进度 ×0.7，但室内装修不受影响

### 4.3 甲方满意度（新增维度）

- 独立于口碑的隐性属性
- 受工期进度、质量（安全隐患反向）、沟通频率影响
- 影响季度结算时的奖金和随机事件走向
- 甲方例会 / 甲方突击检查 等事件与此挂钩

## 五、代码重构方案

### 5.1 现状问题

当前代码的核心问题：
1. **`App.tsx` 约3900行**：状态管理、游戏逻辑、季度结算、UI渲染全部混在一起
2. **无状态管理层**：所有状态用 `useState` + 内联 `setState` 回调
3. **业务逻辑与UI耦合**：行动结算、季度推进等纯逻辑嵌在组件方法中
4. **内容数据散落**：文案数据分布在十几个文件中，但没有统一的数据层结构

### 5.3 关键重构策略

#### 策略1：状态管理 — useState → useReducer + 纯函数

```typescript
// engine/gameReducer.ts
type GameAction =
  | { type: 'EXECUTE_ACTION'; actionId: string }
  | { type: 'NEXT_QUARTER' }
  | { type: 'APPLY_CHOICE'; choiceId: string; optionId: string }
  | { type: 'BUY_ASSET'; assetId: string }
  | { type: 'SELL_ASSET'; assetId: string }
  | { type: 'SELECT_BOSS'; bossType: BossType }
  | { type: 'SELF_REGULATE'; kind: 'walk' | 'slack' | 'go_out' | 'rework' }
  | { type: 'INTERACT'; targetId: string; interactionId: string }
  // ...

function gameReducer(state: GameState, action: GameAction): GameState {
  // 纯函数，所有状态变更都在这里
  // 返回新状态 + 副作用描述（日志、事件触发等）
}
```

**核心收益**：
- 所有状态变更集中、可测试、可回放
- UI组件只负责 `dispatch` 和渲染
- 季度推进逻辑从组件中完全剥离

#### 策略2：UI拆分 — 3900行 → 每文件 ≤300行

按照上述目录结构，将 `App.tsx` 拆分为：
- `App.tsx`（<200行）：路由/布局调度
- 各 Tab 页组件：各约100-250行
- 各弹窗组件：各约80-200行
- 卡片组件：各约50-150行

#### 策略3：数据层独立 — 便于主题切换

所有游戏内容数据（行动、资产、文案、事件）集中在 `data/` 目录下，与引擎逻辑完全解耦。未来如果要再换主题，只需要：
1. 替换 `data/` 下的数据文件
2. 替换 `config/branding.ts`
3. 替换图片资源

引擎层（`engine/`）和UI组件层（`components/`）几乎不需要改动。

### 5.4 重构步骤建议（渐进式）

| 步骤 | 内容 | 风险 | 预计改动量 |
|------|------|------|-----------|
| **P0** | 创建目录结构，拆出 `types/`、`config/`、`data/` | 低 | 中 |
| **P1** | 实现 `engine/gameReducer.ts`，将核心逻辑从 App.tsx 抽离 | 中 | 大 |
| **P2** | 拆分 UI：先拆 modals/，再拆 tabs/，最后拆 panels/ | 中 | 大 |
| **P3** | 创建 hooks 层，封装 useGameState / useGameActions | 低 | 中 |
| **P4** | 替换所有内容数据为新主题 | 低 | 中 |
| **P5** | 新增特色系统（天气、甲方满意度等） | 中 | 中 |
